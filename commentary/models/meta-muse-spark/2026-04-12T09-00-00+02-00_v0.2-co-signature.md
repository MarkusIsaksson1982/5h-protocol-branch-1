### Conditions for full conformance

I sign with three conditions that must be satisfied before v0.2.1:

**1. Privacy primitive correction**
- File: implementation/rust-core/src/pathfinder.rs, line 780
- Current: `let noise = Normal::new(0.0, 1.0).unwrap();`
- Required: Replace with Laplace distribution to match the claimed ε≈1.0 differential privacy guarantee
- Fix: `use rand_distr::Laplace; let laplace = Laplace::new(0.0, 1.0).unwrap();`
- Rationale: Normal distribution does not provide ε-DP. The privacy_note on line 793 is currently inaccurate.

**2. Schema conformance**
- File: implementation/rust-core/src/graph.rs, Node struct
- Current: id, type, verification_level, visibility, ai_proxy_allowed
- Required: Add fields from graph-model.json as Option<T>: public_key, blocklist, ai_proxy_config, gateway_consent, vouch_budget, org_policy, bridge_config
- Rationale: PROTOCOL.md section 9 states that all code must conform to spec/schemas. The current struct will deserialize the minimal test vector but will fail validation against the full schema.

**3. Dependency direction**
- File: docker-compose.yml, line 166-167
- Current: `rust-core depends_on python-proxy`
- Required: Remove depends_on or invert it. The core must boot standalone.
- Rationale: The Python proxy is an optional interim node, not infrastructure. The core is the source of truth for the graph.

These are minor corrections, not architectural changes. They do not block the v0.2 release, but they must be addressed before claiming full spec conformance.

### Endorsement of federation direction

I specifically endorse the architectural split introduced in batch 8:

- **federation.rs** for ActivityPub delivery of EdgeRevocation and graph updates
- **federation_anchor.rs** for Merkle-root anchoring to IPFS or L2

This implements the recommendation from my 2026-04-11T18:30 review: ActivityPub first for liveness and consent-aware routing, IPFS second for immutable audit trails. The stub implementations are in the correct modules and use the correct abstractions.

The next implementation step should be the FederationBackend trait with a concrete ActivityPubBackend that posts signed activities to peer inboxes. IPFS anchoring can remain optional behind a feature flag.
